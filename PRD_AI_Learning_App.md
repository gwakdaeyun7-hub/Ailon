## 📋 Executive Summary

**AI Learning Companion**은 매일 아침 사용자에게 최신 AI 뉴스와 다양한 학문의 기본 원리를 큐레이션하여 제공하고, AI와 학문의 융합을 통해 새로운 아이디어를 생성하는 교육 플랫폼입니다.

### 핵심 가치 제안
- **일일 AI 인사이트**: 매일 아침 선별된 AI 뉴스를 통해 빠르게 변화하는 AI 트렌드를 놓치지 않음
- **점진적 학습**: 복잡한 학문적 개념을 매일 작은 단위로 학습하여 장기적 지식 축적
- **융합적 사고 촉진**: AI와 다른 학문의 원리를 결합하여 혁신적 아이디어 발견

### 비즈니스 목표
1. **비용 효율성**: 월간 운영 비용 $0~$10 이내로 유지 (초기 MVP)
2. **사용자 확보**: 출시 후 3개월 내 MAU 500명 달성
3. **참여도**: 일일 평균 앱 사용 시간 10분 이상, 주간 리텐션 40% 이상

> **💡 핵심 차별점**
> 단순한 뉴스 앱이 아닌, AI와 학문의 융합을 통해 사용자의 창의적 사고를 자극하는 **지식 시너지 플랫폼**

## 🚀 핵심 기능 명세

### Feature 1: 매일 아침 AI 뉴스 큐레이션

**Priority:** Must-Have (P0)
**Effort:** Medium (5 story points)

#### User Story
```
As a AI 트렌드에 관심 있는 사용자,
I want 매일 아침 선별된 AI 뉴스를 받아보고 싶다,
So that 바쁜 일상 속에서도 중요한 AI 동향을 놓치지 않을 수 있다.
```

#### Detailed Requirements

**1.1 자동 뉴스 수집**
- 주요 AI 뉴스 소스에서 자동으로 뉴스 수집
  - TechCrunch, VentureBeat, MIT Technology Review, Ars Technica (RSS 피드)
  - Hacker News API
  - Reddit (r/MachineLearning, r/artificial) API
- 수집 주기: 매일 오전 6시 (KST)
- 수집량: 하루 50~100개 기사

**1.2 뉴스 필터링 및 랭킹**
- AI/ML 관련 키워드 필터링
  - 포함 키워드: AI, ML, Machine Learning, Deep Learning, LLM, GPT, Claude, ChatGPT, Neural Network 등
  - 제외 키워드: 광고성 키워드, 중복 기사
- 중요도 점수 계산 알고리즘
  - 소스 신뢰도 (30%)
  - 참여도 (좋아요, 댓글 수) (30%)
  - 키워드 관련성 (20%)
  - 신선도 (최근성) (20%)
- 상위 5~7개 기사 선정

**1.3 요약 생성**
- 각 기사당 2~3문장 요약 생성
- 무료 LLM API 활용 (Gemini Flash, Groq 등)
- Fallback: 기사 첫 2~3문장 추출

**1.4 사용자 인터페이스**
- 카드 형식으로 뉴스 표시
  - 제목, 요약, 출처, 링크, 발행 시간
- 카테고리 태그 표시 (연구, 제품, 비즈니스, 윤리 등)
- "더 읽기" 버튼으로 원문 링크 연결

#### Acceptance Criteria

```gherkin
Given 사용자가 오전 7시에 앱을 열었을 때,
When 홈 화면에 접속하면,
Then 최신 AI 뉴스 5~7개가 요약과 함께 표시되어야 한다.

Given 뉴스 카드가 표시되었을 때,
When 사용자가 "더 읽기"를 클릭하면,
Then 원문 기사가 새 브라우저 탭에서 열려야 한다.

Given 뉴스 수집 프로세스가 실패했을 때,
When 시스템이 에러를 감지하면,
Then 이전 날짜의 뉴스를 표시하고 에러 메시지를 로그에 기록해야 한다.

Given 사용자가 특정 카테고리에 관심이 있을 때,
When 카테고리 필터를 선택하면,
Then 해당 카테고리의 뉴스만 필터링되어 표시되어야 한다.
```

#### Technical Specifications
- RSS 파싱: Python `feedparser` 라이브러리
- API 호출: Python `requests`
- 요약 생성: Google Gemini Flash API (무료 티어) 또는 Groq API
- 스케줄링: GitHub Actions (Cron job) 또는 Railway (무료 티어)
- 데이터 저장: Firebase Firestore (무료 티어) 또는 Supabase (무료 티어)

#### Dependencies
- Google Gemini API 키 발급
- RSS 피드 소스 목록 확정
- 백엔드 스케줄러 설정

### Feature 2: 학문 기본 원리 일일 학습

**Priority:** Must-Have (P0)
**Effort:** High (8 story points)

#### User Story
```
As a 지적 호기심이 많은 학습자,
I want 매일 다양한 학문의 기본 원리를 짧게 학습하고 싶다,
So that 폭넓은 교양 지식을 체계적으로 쌓을 수 있다.
```

#### Detailed Requirements

**2.1 학문 카테고리 정의**

초기 지원 카테고리 (MVP):
- 물리학 (Physics)
- 화학 (Chemistry)
- 생물학 (Biology)
- 철학 (Philosophy)
- 경제학 (Economics)
- 심리학 (Psychology)

향후 확장 카테고리:
- 수학, 사회학, 정치학, 역사학, 문학, 예술 등

**2.2 콘텐츠 구조**

각 학습 콘텐츠는 다음 구조를 따름:
1. **제목**: 원리/개념 이름
2. **카테고리**: 학문 분야
3. **난이도**: 입문 / 중급 / 고급
4. **설명**: 300~500자 이내 설명
5. **예시**: 실생활 적용 예시
6. **관련 개념**: 연결된 다른 원리 링크
7. **출처/참고**: 더 깊이 학습할 수 있는 자료 링크

예시:
```
제목: 엔트로피 (Entropy)
카테고리: 물리학
난이도: 입문

설명: 엔트로피는 시스템의 무질서도를 나타내는 물리량입니다.
열역학 제2법칙에 따르면, 고립된 시스템의 엔트로피는 시간이
지날수록 증가하는 경향이 있습니다. 이는 자연 현상이 자발적으로
무질서한 방향으로 진행됨을 의미합니다.

예시: 방을 정리하지 않으면 자연스럽게 어질러지는 현상,
뜨거운 커피가 식으면서 주변 공기와 같은 온도가 되는 현상 등이
엔트로피 증가의 예입니다.

관련 개념: 열역학 제2법칙, 가역/비가역 과정, 통계역학
```

**2.3 콘텐츠 소싱 전략**

**옵션 A: 수동 큐레이션 (초기 MVP)**
- 프로덕트 팀이 직접 작성
- Wikipedia, Khan Academy, Stanford Encyclopedia of Philosophy 등 참고
- 초기 100개 콘텐츠 준비 (약 3개월치)

**옵션 B: AI 생성 + 검토 (장기)**
- LLM으로 초안 생성 (Gemini, Claude API)
- 사람이 팩트 체크 및 편집
- 품질 관리 프로세스 구축

**옵션 C: 오픈소스 콘텐츠 활용**
- Wikipedia API 활용 (요약 섹션 파싱)
- Creative Commons 라이선스 콘텐츠 큐레이션
- 출처 명시 및 라이선스 준수

**권장 방식 (MVP)**: 옵션 A + C 조합
- 초기 30개는 수동 작성하여 품질 기준 확립
- 이후 Wikipedia API + LLM으로 확장
- 모든 콘텐츠는 팀 검토 후 게시

**2.4 일일 제공 로직**
- 매일 1개의 원리를 사용자에게 제공
- 순차적 제공: 사용자가 이전 내용을 본 다음 원리 제공
- 카테고리 균형: 일주일에 각 카테고리가 최소 1회씩 등장
- 난이도 조절: 입문 → 중급 → 고급 순서로 점진적 난이도 상승

**2.5 사용자 인터페이스**
- 일일 학습 섹션에 "오늘의 원리" 카드 표시
- 카테고리 아이콘 및 색상 코딩
- 예상 읽기 시간 표시 (약 2~3분)
- 북마크 기능
- 이전 학습한 원리 히스토리 조회 가능
- 카테고리별 필터링

#### Acceptance Criteria

```gherkin
Given 사용자가 앱에 로그인했을 때,
When 학습 섹션에 접속하면,
Then 오늘의 원리 1개가 표시되어야 한다.

Given 사용자가 오늘의 원리를 읽었을 때,
When 읽기 완료 표시를 하면,
Then 학습 기록에 저장되고 다음 날 새로운 원리가 제공되어야 한다.

Given 사용자가 특정 카테고리만 보고 싶을 때,
When 카테고리 필터를 적용하면,
Then 해당 카테고리의 원리만 순차적으로 제공되어야 한다.

Given 사용자가 이전에 학습한 내용을 다시 보고 싶을 때,
When 히스토리 페이지에 접속하면,
Then 지금까지 학습한 모든 원리가 날짜순으로 표시되어야 한다.

Given 사용자가 흥미로운 원리를 발견했을 때,
When 북마크 버튼을 클릭하면,
Then 북마크 목록에 저장되어야 한다.
```

#### Technical Specifications
- 콘텐츠 저장: JSON 파일 또는 Firebase Firestore
- Wikipedia API: Python `wikipedia` 라이브러리
- 콘텐츠 생성: Google Gemini API (검토 필요)
- 사용자 학습 기록: Firebase Authentication + Firestore
- 프론트엔드: React 또는 Next.js

#### Dependencies
- 초기 콘텐츠 100개 작성 완료
- 콘텐츠 스키마 정의
- Wikipedia API 통합 (선택사항)

### Feature 3: AI-학문 융합 아이디어 생성

**Priority:** Must-Have (P0)
**Effort:** High (8 story points)

#### User Story
```
As a 창의적인 사고를 하고 싶은 사용자,
I want AI와 다양한 학문의 원리를 융합한 아이디어를 받아보고 싶다,
So that 새로운 관점과 영감을 얻어 혁신적인 생각을 할 수 있다.
```

#### Detailed Requirements

**3.1 아이디어 생성 로직**

**Input:**
- 오늘의 AI 뉴스 (1~2개 선택)
- 오늘의 학문 원리 (1개)
- 선택적: 사용자 관심 분야/직업 정보

**Process:**
1. AI 뉴스에서 핵심 기술/트렌드 추출
2. 학문 원리의 핵심 개념 추출
3. LLM 프롬프트로 융합 아이디어 생성
4. 생성된 아이디어 구조화

**Output 구조:**
```
제목: [간결한 아이디어 제목]
융합: [AI 기술] × [학문 원리]

아이디어:
[2~3문장으로 아이디어 설명]

적용 가능 분야:
- 분야 1
- 분야 2
- 분야 3

실현 가능성: [높음/중간/낮음]
혁신도: [⭐⭐⭐⭐⭐] (5점 척도)
```

**3.2 아이디어 생성 프롬프트 템플릿**

```
당신은 창의적인 아이디어 생성 전문가입니다.

오늘의 AI 뉴스:
{news_summary}

오늘의 학문 원리:
{principle_title} ({discipline})
{principle_summary}

위 AI 기술/트렌드와 학문 원리를 융합하여 혁신적인 아이디어를
생성해주세요. 아이디어는 다음 조건을 만족해야 합니다:

1. AI 기술과 학문 원리가 명확하게 결합되어야 함
2. 실용적이거나 사고를 자극하는 내용이어야 함
3. 구체적인 적용 분야를 제시해야 함

출력 형식:
제목: [아이디어 제목]
융합: [AI 기술] × [학문 원리]
아이디어: [설명]
적용 가능 분야: [3개]
실현 가능성: [높음/중간/낮음]
```

**3.3 예시**

```
제목: 엔트로피 기반 AI 학습 최적화

융합: 강화학습 × 엔트로피 (물리학)

아이디어:
강화학습 에이전트가 환경을 탐색할 때, 엔트로피 개념을
적용하여 탐색의 다양성을 최대화할 수 있습니다.
높은 엔트로피 상태를 보상으로 설정하면, 에이전트는
덜 방문한 상태를 우선 탐색하여 더 효율적으로 학습합니다.

적용 가능 분야:
- 게임 AI 개발
- 로봇 경로 탐색
- 추천 시스템 다양성 개선

실현 가능성: 높음
혁신도: ⭐⭐⭐⭐
```

**3.4 아이디어 품질 관리**
- 생성된 아이디어 자동 평가 (관련성, 명확성, 창의성)
- 낮은 품질 아이디어는 재생성 (최대 3회 시도)
- 사용자 피드백 수집 (좋아요/싫어요)
- 피드백 데이터로 프롬프트 개선

**3.5 사용자 인터페이스**
- 일일 1개의 융합 아이디어 제공
- 아이디어 카드 형태로 표시
- "다른 아이디어 보기" 버튼 (재생성, 하루 3회 제한)
- 아이디어 저장/공유 기능
- 아이디어 히스토리 조회
- 좋아요/북마크 기능

**3.6 고급 기능 (Phase 2)**
- 사용자 관심사 기반 맞춤형 아이디어
- 여러 학문 원리 조합 (2개 이상)
- 아이디어 상세 확장 (클릭 시 더 깊은 설명)
- 커뮤니티에서 다른 사용자 아이디어 보기

#### Acceptance Criteria

```gherkin
Given 사용자가 오늘의 뉴스와 원리를 확인했을 때,
When 아이디어 섹션에 접속하면,
Then AI와 학문 원리를 융합한 아이디어 1개가 표시되어야 한다.

Given 아이디어가 표시되었을 때,
When 아이디어를 읽으면,
Then 제목, 융합 요소, 설명, 적용 분야, 실현 가능성이 모두 포함되어야 한다.

Given 사용자가 아이디어가 마음에 들지 않을 때,
When "다른 아이디어 보기"를 클릭하면,
Then 새로운 아이디어가 생성되어 표시되어야 한다 (하루 최대 3회).

Given 사용자가 흥미로운 아이디어를 발견했을 때,
When 북마크 버튼을 클릭하면,
Then 북마크 목록에 저장되어야 한다.

Given 사용자가 아이디어를 평가할 때,
When 좋아요 또는 싫어요 버튼을 클릭하면,
Then 피드백이 시스템에 저장되고 향후 아이디어 개선에 반영되어야 한다.
```

#### Technical Specifications
- 아이디어 생성: Google Gemini Flash API 또는 Groq Llama
- 프롬프트 관리: LangChain 또는 직접 구현
- 캐싱: 생성된 아이디어는 24시간 캐싱 (비용 절감)
- 사용자 피드백 저장: Firebase Firestore
- Rate Limiting: 하루 아이디어 재생성 3회 제한

#### Dependencies
- LLM API 선택 및 설정
- 프롬프트 엔지니어링 및 테스트
- 뉴스 및 원리 데이터 파이프라인 완성

### Feature 4: 사용자 인증 및 개인화

**Priority:** Should-Have (P1)
**Effort:** Medium (5 story points)

#### User Story
```
As a 지속적으로 앱을 사용하는 사용자,
I want 내 학습 기록과 선호도를 저장하고 싶다,
So that 개인화된 경험을 받고 학습 진행 상황을 추적할 수 있다.
```

#### Detailed Requirements

**4.1 인증 방식**
- 이메일/비밀번호 로그인
- Google 소셜 로그인
- 익명 사용 모드 (기능 제한)

**4.2 개인화 기능**
- 관심 학문 분야 선택 (온보딩 시)
- 관심 AI 카테고리 선택 (연구/제품/비즈니스 등)
- 알림 시간 설정 (매일 아침 몇 시에 받을지)
- 언어 설정 (한국어/영어)

**4.3 학습 기록**
- 읽은 뉴스 기록
- 학습한 원리 기록
- 생성된 아이디어 기록
- 북마크 목록
- 학습 연속 일수 (Streak) 추적

**4.4 프로필 페이지**
- 사용자 정보
- 학습 통계 (총 학습 일수, 읽은 뉴스 수, 학습한 원리 수)
- 학습 캘린더 (히트맵)
- 설정 변경

#### Acceptance Criteria

```gherkin
Given 신규 사용자가 앱을 처음 열었을 때,
When 온보딩 화면이 표시되면,
Then 관심 분야를 선택하고 계정을 생성할 수 있어야 한다.

Given 사용자가 로그인했을 때,
When 여러 기기에서 접속하면,
Then 모든 기기에서 동일한 학습 기록이 동기화되어야 한다.

Given 사용자가 학습을 진행할 때,
When 뉴스를 읽거나 원리를 학습하면,
Then 학습 기록이 자동으로 저장되어야 한다.
```

#### Technical Specifications
- 인증: Firebase Authentication (무료 티어)
- 데이터 저장: Firebase Firestore
- 소셜 로그인: Google OAuth

### Feature 5: 푸시 알림 및 스케줄링

**Priority:** Should-Have (P1)
**Effort:** Low (3 story points)

#### User Story
```
As a 바쁜 일상을 사는 사용자,
I want 매일 아침 정해진 시간에 알림을 받고 싶다,
So that 학습 습관을 유지하고 앱을 잊지 않을 수 있다.
```

#### Detailed Requirements

**5.1 알림 유형**
- 일일 학습 알림: "오늘의 AI 뉴스와 학문 원리가 준비되었습니다!"
- 연속 학습 격려: "3일 연속 학습 중입니다! 계속 유지하세요!"
- 주간 요약: "이번 주 배운 내용을 복습해보세요."

**5.2 알림 설정**
- 알림 시간 커스터마이징 (기본: 오전 7시)
- 알림 On/Off
- 알림 유형별 On/Off

#### Acceptance Criteria

```gherkin
Given 사용자가 알림을 활성화했을 때,
When 설정한 시간이 되면,
Then 푸시 알림을 받아야 한다.

Given 사용자가 알림을 비활성화했을 때,
When 설정한 시간이 되어도,
Then 알림을 받지 않아야 한다.
```

#### Technical Specifications
- 웹: Web Push API 또는 Firebase Cloud Messaging (무료)
- 모바일: Firebase Cloud Messaging
- 스케줄링: Firebase Cloud Functions (무료 티어)

## 💰 비용 최적화 전략

### 무료 티어 최대 활용

#### Firebase (무료 티어 제한)
**제공량:**
- Firestore: 1GB 저장, 50K 읽기/일, 20K 쓰기/일
- Cloud Functions: 2M 호출/월
- Hosting: 10GB 저장, 360MB/일 전송
- Authentication: 무제한

**최적화 전략:**
1. **읽기/쓰기 최소화**
   - 클라이언트 측 캐싱 적극 활용 (React Query, SWR)
   - 오늘의 데이터만 자주 읽기, 과거 데이터는 필요시에만
   - 배치 쓰기 사용 (여러 문서를 한 번에)

2. **데이터 구조 최적화**
   - 중복 제거, 정규화된 스키마
   - 큰 배열은 별도 문서로 분리
   - 인덱스 최소화

3. **예상 사용량 (500 MAU 기준)**
   - 읽기: 500명 × 10 읽기/일 = 5,000 읽기/일 ✅ (제한: 50K)
   - 쓰기: 500명 × 5 쓰기/일 = 2,500 쓰기/일 ✅ (제한: 20K)
   - **여유 있음**

---

#### LLM API (무료 티어)

**Google Gemini Flash:**
- 15 RPM (분당 요청)
- 1M TPM (분당 토큰)
- 1,500 RPD (일일 요청)

**Groq Llama 3:**
- 14,400 요청/일
- 매우 빠른 응답 속도

**최적화 전략:**
1. **캐싱 적극 활용**
   - 생성된 요약/아이디어를 24시간 캐싱
   - 동일한 뉴스는 재요약하지 않음

2. **배치 처리**
   - 여러 뉴스를 한 번의 API 호출로 요약
   - 프롬프트 최적화로 토큰 사용량 감소

3. **폴백 전략**
   - API 한도 초과 시 간단한 규칙 기반 요약으로 대체
   - 여러 API 로테이션 (Gemini → Groq → OpenRouter)

4. **예상 사용량 (500 MAU 기준)**
   - 뉴스 요약: 7개 × 1회/일 = 7 요청/일
   - 아이디어 생성: 500명 × 평균 2회/일 = 1,000 요청/일
   - **총: 약 1,000 요청/일** ✅ (Gemini 제한: 1,500, Groq 제한: 14,400)

---

#### Vercel 호스팅 (무료 티어)
**제공량:**
- 100GB 대역폭/월
- 무제한 배포

**최적화 전략:**
1. 이미지 최적화 (Next.js Image 컴포넌트)
2. 정적 페이지 최대 활용 (SSG)
3. CDN 캐싱

**예상 사용량 (500 MAU):**
- 평균 페이지 크기: 500KB
- 500명 × 30 방문/월 × 500KB = 7.5GB/월 ✅ (제한: 100GB)

---

#### GitHub Actions (무료 티어)
**제공량:**
- Public repo: 무제한
- Private repo: 2,000분/월

**최적화 전략:**
1. Public repo 사용
2. 스크립트 실행 시간 최소화 (10분 이내)

**예상 사용량:**
- 뉴스 수집: 1회/일 × 10분 = 300분/월 ✅
