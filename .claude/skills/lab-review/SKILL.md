# Lab Review -- 시뮬레이션 품질 검토 및 수정

LAB 탭의 시뮬레이션 코드(`mobile/components/snaps/simulations/*.ts`)가 사용자의 **학문 원리 이해에 실제로 도움이 되는지** 검토하고, 교육적 퀄리티가 높은 시뮬레이션으로 수정합니다.

> **텍스트 콘텐츠(`scripts/curated_principles/*.md`)는 읽기 전용**으로만 참조한다.
> 시뮬레이션이 텍스트와 잘 맞는지 확인하는 용도로만 사용하며, 텍스트 자체는 수정하지 않는다.

> **principle-qa와의 차이**: `principle-qa`는 텍스트 콘텐츠의 구조/형식/길이 등을 검사합니다.
> `lab-review`는 **시뮬레이션 코드 품질** — 교육 효과, UI/UX, 렌더링, 테마/다국어 호환 — 에 집중합니다.

## 사용법

### 기본 명령

- `/lab-review` -- 미검토 원리 자동 선택 (1개, 우선순위 기반)
- `/lab-review opt_gradient_descent` -- 특정 원리 검토
- `/lab-review all` -- 전체 일괄 검토 (리포트만)
- `/lab-review fix neuro_attention_mechanism` -- 검토 없이 바로 수정

### 추가 명령

- `/lab-review scan` -- 빠른 스캔: 전체 시뮬레이션 렌더링 호환성 + 테마/다국어 체크 (에이전트 없이 1분 내)
- `/lab-review compare {seed_id}` -- SA 시뮬레이션과 나란히 비교: 시뮬레이션 품질 차원별 대조
- `/lab-review dashboard` -- 검토 현황: 등급 분포, 미검토 목록, 우선순위 큐
- `/lab-review sim-audit` -- 시뮬레이션 필요성 일괄 평가 (SEED_TO_SIM 미보유 원리)

검토 결과는 `.claude/lab-review-state.json`에 누적 저장한다.

**초기화**: 파일이 없으면 첫 실행 시 아래 구조로 자동 생성한다. SA는 Gold Standard이므로 A등급으로 사전 등록한다.

```json
      "notes": "Gold Standard — 수정 불가"
    }
  },
  "lastFullScan": null,
  "lastDashboard": null
}
```

### 자동 우선순위 선택 (`/lab-review` 인자 없음)

다음 순서로 1개를 자동 선택한다:
1. **미검토** 원리 (state.json에 없는 것, SEED_TO_SIM 보유 원리 중)
2. **C 등급** 원리 (가장 오래된 검토 우선)
3. **B 등급** 원리 (가장 오래된 검토 우선)
4. 모두 A이면 "전체 A 등급 — 검토 불필요" 메시지

### `/lab-review scan` -- 빠른 스캔 모드

에이전트 없이 직접 수행하는 경량 검사. SEED_TO_SIM 보유 시뮬레이션 전체를 1분 내 스캔한다.

**검사 항목:**
1. JavaScript/HTML 구문 오류 가능성 (self-contained 여부)
2. 외부 의존성 참조 여부
3. 다크/라이트 테마 하드코딩 색상(`#fff`, `#000` 등) 감지
4. KO/EN 레이블 양쪽 존재 여부
5. 터치 타겟 크기 (44pt 미만 감지)
6. WebView 내부/외부 스크롤 충돌 가능성

**출력:** 문제 있는 시뮬레이션만 테이블로 표시. 전부 통과 시 "SCAN PASS".

## Gold Standard: SA 시뮬레이션이 우수한 이유

**참조 파일:** `mobile/components/snaps/simulations/mpc.ts` (SA 시뮬레이션)
**텍스트 참조:** `scripts/curated_principles/opt_simulated_annealing.md` (읽기 전용)

SA 시뮬레이션이 교육적으로 우수한 이유를 다음 5가지로 분석한다. 모든 시뮬레이션은 이 5가지를 갖춰야 한다.

### 1. 렌더링 정상성

SA 시뮬레이션은 외부 의존성 없이 self-contained으로 동작한다. WebView 환경에서 JavaScript/HTML 구문 오류 없이 렌더링된다.

**체크포인트:**
- 외부 CDN/라이브러리 의존 없이 순수 JS+HTML+CSS로 동작하는가?
- 런타임 에러 가능성이 없는가?
- Canvas/SVG 등 렌더링 요소가 정상적으로 표시되는가?

### 2. UI 직관성 (슬라이더/버튼)

SA의 파라미터 컨트롤은 레이블이 명확하고, 슬라이더 범위가 의미 있는 변화를 보여주며, 초기값이 교육적으로 좋은 상태다.

**체크포인트:**
- 레이블이 전문 용어가 아닌 이해 가능한 표현인가?
- 슬라이더 범위/스텝이 적절한가? (의미 있는 변화가 보이는 범위)
- 초기값이 교육적으로 의미 있는 상태인가?
- 조작 후 피드백이 즉각적인가? (지연 없이 시각 반영)

### 3. 교육 효과 (원리 이해 기여)

SA 시뮬레이션은 텍스트에서 설명한 핵심 메커니즘(온도 하강 → 탐색 범위 축소)을 파라미터 조절로 직접 체험하게 한다. 단순 데모가 아닌 "아하" 모먼트를 유발한다.

**체크포인트:**
- 텍스트 콘텐츠에서 설명한 핵심 메커니즘이 시뮬레이션에서 체험 가능한가?
- 파라미터 변화 → 결과 변화의 인과관계가 시각적으로 명확한가?
- 단순 데모가 아닌, 사용자가 "아하" 모먼트를 경험할 수 있는가?
- 시뮬레이션에서만 이해할 수 있는 추가 인사이트가 있는가?

### 4. 테마/다국어 호환성

SA는 CSS 변수 또는 테마 파라미터를 사용하여 다크/라이트 모드 모두 정상이고, KO/EN 레이블이 완비되어 있다.

**체크포인트:**
- 다크/라이트 테마 모두에서 배경색, 텍스트, 그래프 색상이 가독성 확보되는가?
- 하드코딩된 색상(`#fff`, `#000` 등)이 테마 전환 시 깨지지 않는가?
- CSS 변수 또는 테마 파라미터를 올바르게 사용하는가?
- KO/EN 레이블, 설명 텍스트가 양쪽 언어로 제공되는가?
- 언어 전환 시 레이아웃이 깨지지 않는가? (CJK vs Latin 길이 차이)

### 5. Snaps-Lab 상호 보완성

SA 시뮬레이션은 텍스트만으로는 전달하기 어려운 "온도에 따른 수용 확률 변화"를 시각적으로 보여준다. 텍스트와 시뮬레이션이 상호 보완적이다.

**체크포인트:**
- 텍스트에서 설명한 개념을 시뮬레이션이 보강하는가?
- 시뮬레이션에서만 이해할 수 있는 추가 인사이트가 있는가?
- 텍스트와 시뮬레이션 간 용어/표현이 일관되는가?

## 검토 프로세스: 에이전트 배분

### Phase 1: 시뮬레이션 품질 검토 (simulation-expert + app-developer) -- 시뮬레이션 보유 원리만

SEED_TO_SIM에 **있는** 원리에 대해서만 실행한다. 시뮬레이션 코드 자체의 품질과 교육 효과를 검증한다.

> **실행 전**: `mobile/lib/labPrinciples.ts`의 `SEED_TO_SIM`을 읽어 현재 시뮬레이션 보유 원리 목록을 확인한다. 대상 원리가 목록에 없으면 이 Phase를 건너뛴다.
> **텍스트 참조**: `scripts/curated_principles/{seed_id}.md`를 읽어 텍스트 콘텐츠를 확인한다. 시뮬레이션이 텍스트와 잘 맞는지 비교 용도로만 사용한다.

**Step 1: simulation-expert + app-developer 병렬 실행**

두 에이전트를 **동시에** 호출한다.

**simulation-expert에게:**

```

3. 텍스트 콘텐츠(Snaps)와 시뮬레이션(Lab)이 상호 보완되는가?
   - 텍스트에서 설명한 개념을 시뮬레이션이 보강하는가?
   - 시뮬레이션에서만 이해할 수 있는 추가 인사이트가 있는가?
   - 텍스트와 시뮬레이션 간 용어/표현이 일관되는가?

각 항목에 A/B/C 등급 + 구체적 개선 제안을 제시해줘.
```

**app-developer에게:**

```
다음 학문원리 시뮬레이션의 UI/UX 품질을 검토해줘:

검토 기준:
1. 파라미터 슬라이더/버튼이 직관적인가?

   - 터치 타겟이 충분한 크기인가? (최소 44pt)
   - 스크롤 충돌이 없는가? (WebView 내부 스크롤 vs 외부 스크롤)

각 항목에 A/B/C 등급 + 구체적 개선 제안을 제시해줘.
```

### Phase 2: 시뮬레이션 필요성 평가 (simulation-expert + app-developer) -- 시뮬레이션 미보유 원리만

시뮬레이션이 없는 원리(SEED_TO_SIM에 없는 원리, 현재 20개)에 대해 **simulation-expert** 에이전트로 교육적 임팩트를 평가하고, 필요 시 **app-developer**로 기술 구현 가능성을 보조 확인한다.

> **실행 전**: `mobile/lib/labPrinciples.ts`의 `SEED_TO_SIM`을 읽어 현재 시뮬레이션 보유 원리 목록을 확인한다. 하드코딩된 목록을 사용하지 않는다.
> **텍스트 참조**: `scripts/curated_principles/{seed_id}.md`를 읽어 텍스트 콘텐츠를 확인한다. 시뮬레이션이 어떤 개념을 시각화해야 하는지 파악하는 용도.

**Step 1: 교육적 임팩트 평가 (simulation-expert)**

```
다음 학문원리에 인터랙티브 시뮬레이션이 필요한지 평가해줘:
원리: {principle_name}
콘텐츠 요약: {one_line_summary}

현재 시뮬레이션이 있는 원리 (SEED_TO_SIM에서 동적으로 읽은 목록):
{SEED_TO_SIM 키 목록을 여기에 나열}

평가 기준:
1. 시뮬레이션 없이 텍스트만으로 핵심 메커니즘을 이해할 수 있는가?
2. 파라미터 조절이 결과에 미치는 영향을 체험하는 것이 교육적 가치가 높은가?
3. 시각적 피드백(그래프, 애니메이션)이 텍스트 설명보다 효과적인 개념이 있는가?
4. 시뮬레이션 추가 시 사용자 이해도 향상 폭을 예측해줘 (what-if 시나리오).
5. 기존 시뮬레이션(SA, GD 등)과 비교해 구현 대비 교육 효과 ROI를 판단해줘.

결론: ESSENTIAL(필수) / BENEFICIAL(도움됨) / OPTIONAL(텍스트 충분) 중 하나로 판정.
ESSENTIAL/BENEFICIAL이면 어떤 시각화가 효과적일지 구체적으로 제안해줘.
```

**Step 2 (ESSENTIAL/BENEFICIAL만): 기술 구현 가능성 확인 (app-developer)**

simulation-expert가 ESSENTIAL/BENEFICIAL로 판정한 경우에만 app-developer에게 React Native 환경에서의 구현 가능성과 기술적 제약을 확인한다.

> **주의**: 시뮬레이션 개발은 이 스킬의 범위가 아니다. 필요성 평가와 제안만 수행한다.

### Phase 3: 렌더링 호환성 (자동)

직접 수행한다 (에이전트 불필요). 시뮬레이션 코드의 WebView 렌더링 호환성을 검사한다:

- self-contained 여부 (외부 CDN/라이브러리 참조 없는지)
- JavaScript 구문 오류 가능성
- Canvas/SVG 렌더링 요소 정상 여부
- CSS 변수 사용 패턴 (테마 호환)
- 터치 이벤트 핸들링 (스크롤 충돌)

### Phase 4: 수정 (B 이하만)

시뮬레이션 수정이 필요하면 에이전트를 활용한다:

| 수정 유형 | 에이전트 |
|-----------|---------|
| 시뮬레이션 교육 효과/메커니즘 수정 | simulation-expert |
| 시뮬레이션 UI/테마/다국어 수정 | app-developer |
| 렌더링 호환성 수정 | 직접 수행 |

**수정 원칙:**
- SA 시뮬레이션을 참조하되 복사 금지
- A등급 차원은 보존 (좋은 부분을 망가뜨리지 않기)
- 다크/라이트 테마 양쪽 확인
- KO/EN 레이블 양쪽 확인
- 외부 의존성 없이 self-contained 유지
- **opt_simulated_annealing 시뮬레이션은 절대 수정 금지** -- Gold Standard 참조 전용
- **텍스트 콘텐츠(`scripts/curated_principles/*.md`)는 절대 수정 금지** -- 참조 전용

## 에이전트 호출 가이드

### 병렬 실행

Phase 1(simulation-expert + app-developer)은 두 에이전트를 **병렬 실행**한다. 결과를 종합하여 수정 여부를 판단한다.

### 에이전트 프롬프트에 포함할 컨텍스트

모든 에이전트 호출에 다음을 포함한다:
- 시뮬레이션 코드 전문
- 텍스트 콘텐츠 요약 (시뮬레이션-텍스트 정합성 확인 용도)
- SA Gold Standard 시뮬레이션에서 해당 차원의 구체적 예시

### 사용 가능한 에이전트 목록

| 에이전트 | 역할 | 사용 시점 |
|---------|------|----------|
| simulation-expert | 시뮬 품질 검토 (보유), 필요성 평가 (미보유), what-if, ROI | Phase 1 (보유), Phase 2 (미보유), sim-audit, Phase 4 수정 |
| app-developer | 시뮬 UI/테마/다국어 검토 (보유), 구현 가능성 (미보유) | Phase 1 (보유), Phase 2 (보조), Phase 4 수정 |
| mobile-frontend-dev | 렌더링 관련 이슈 디버그 | Phase 3 이슈 발견 시 |

## 수정 후 검증 체크리스트

시뮬레이션 수정 완료 후 반드시 확인:

1. **다크/라이트 테마**: 양쪽 모드에서 색상, 가독성 정상인가?
2. **KO/EN 레이블**: 양쪽 언어 레이블이 모두 존재하고 자연스러운가?
3. **self-contained**: 외부 의존성(CDN, 라이브러리) 없이 독립 동작하는가?
4. **터치 UX**: 터치 타겟 44pt 이상, 스크롤 충돌 없는가?
5. **파라미터 범위**: 슬라이더 범위/스텝이 의미 있는 변화를 보여주는가?
6. **state.json 업데이트**: 수정 후 등급과 날짜를 `.claude/lab-review-state.json`의 `simGrade` 필드에 반영했는가?

## 주의사항

- **텍스트 콘텐츠(`scripts/curated_principles/*.md`)는 절대 수정하지 않는다** -- 참조 전용
- Phase 1은 시뮬레이션 보유 원리만 (simulation-expert + app-developer 병렬), Phase 2는 미보유 원리만 (simulation-expert → 판정 후 app-developer 보조)
- 일괄 시 super_category별로 분할 실행 (공학 → 자연과학 → 형식과학 → 응용과학)
- 수정은 기존 시뮬레이션 파일 Edit, 새 파일 생성 금지
- 종합 A 등급은 리포트만 제공, 수정하지 않음
- scan 모드로 먼저 전체 현황 파악 후 개별 심층 검토를 권장
- state.json은 `.gitignore` 대상 -- 검토 상태는 로컬 전용
